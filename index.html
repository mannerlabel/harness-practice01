🔓 Module 01 — 로그인 우회 (Login Bypass)
취약한 로그인 폼에 SQL 구문을 삽입해 인증을 우회합니다.
아래 입력창에 공격 페이로드를 입력하고 어떤 결과가 발생하는지 확인하세요.
vulnerable_login.php
공격 예시: 클릭해서 채우기 → ' OR '1'='1
server_response
실행된 SQL 쿼리
// 실행 버튼을 누르면 쿼리가 표시됩니다
결과
// 실행 결과가 여기 표시됩니다

⚙️ 어떻게 우회가 되는 걸까?

취약한 코드는 입력값을 그대로 쿼리 문자열에 붙입니다:

WHERE id='{입력값}' AND pw='{입력값}'

공격자가 ' OR '1'='1을 입력하면, 최종 쿼리는 다음과 같이 변형됩니다:

WHERE id='' OR '1'='1' AND pw='...'

'1'='1'은 항상 참(TRUE)이므로 WHERE 조건 전체가 참이 되어 모든 레코드가 반환됩니다. 서버는 첫 번째 레코드(보통 관리자)로 로그인을 허용합니다.

server_response
실행된 SQL 쿼리
// 실행 버튼을 누르면 쿼리가 표시됩니다
결과
// 실행 결과가 여기 표시됩니다

⚙️ UNION Injection 원리

UNION은 두 SELECT 결과를 합치는 SQL 명령입니다. 원래 쿼리:

SELECT name, price FROM products WHERE name='노트북'

공격 페이로드 ' UNION SELECT id, password FROM users--를 삽입하면:

SELECT name, price FROM products WHERE name='' UNION SELECT id, password FROM users--'

--는 SQL 주석 처리로 이후 구문을 무력화합니다. 결과 컬럼 수가 일치하면 users 테이블 전체가 결과에 포함됩니다.

⚙️ Prepared Statement가 안전한 이유

취약한 코드는 입력값을 쿼리 문자열에 직접 붙이므로 SQL 구조 자체가 변형됩니다.

Prepared Statement는 쿼리 구조를 먼저 컴파일하고, 입력값은 나중에 ? 자리에 데이터로만 바인딩합니다.

따라서 ' OR '1'='1은 SQL 연산자가 아닌 단순 문자열 리터럴로 처리되어, id = "' OR '1'='1"인 사용자를 찾게 됩니다 — 그런 사용자는 존재하지 않으므로 로그인 실패.

핵심: 데이터와 코드를 명확히 분리하는 것이 SQL Injection 방어의 본질입니다.
